# Transaction Semantics

**Status:** Accepted for Phase 0
**Decision date:** 2026-07-24
**Tracker:** P0-002

## Outcome

Define deterministic local editing semantics before selection, collaboration,
history persistence, or DOCX editing depends on them.

Phase 0 supports:

- text insertion;
- deletion inside one paragraph;
- paragraph splitting;
- joining adjacent paragraphs;
- position mapping for every supported operation;
- inverse generation;
- session-local undo and redo.

Cross-paragraph range deletion, node movement, properties, tables, annotations,
and collaborative rebasing require later design records.

## Revision Contract

- a new session begins at revision zero;
- every successful forward, undo, or redo transaction increments revision once;
- failed transactions do not change document, revision, mapping, or history;
- a transaction declares one base revision;
- a mismatched base revision returns `ODC-2001`;
- revision overflow is fatal and does not partially commit.

Revision is session-local. It is not a wall-clock timestamp, persistence version,
or collaboration vector.

## Positions and Ranges

A text position is:

```text
Position {
  node: NodeId,
  grapheme_offset: u32,
  affinity: Before | After
}
```

A range has ordered `start` and `end` positions. Phase 0 deletion requires both
positions to address the same paragraph and `start <= end`. Empty ranges are
rejected as ineffective transactions.

Positions always resolve against the state produced by prior operations in the
same transaction.

## Operations

### Insert Text

```text
InsertText { at, text, marks }
```

Preconditions:

- `at` resolves to a paragraph grapheme boundary;
- text is non-empty;
- text contains no NUL, tab, paragraph, or line-breaking control;
- resulting paragraph length remains addressable.

The inserted run is normalized with equivalent neighboring runs.

### Delete Range

```text
DeleteRange { range }
```

Preconditions:

- start and end resolve in the same paragraph;
- start is strictly before end;
- both offsets are valid grapheme boundaries.

Deletion retains exact text and mark runs in the generated inverse operation.

### Split Paragraph

```text
SplitParagraph { at, new_id }
```

The original paragraph keeps content before the boundary. A new paragraph with
`new_id` is inserted immediately after it and receives content after the
boundary. The new ID must be non-zero and unique in the document.

At offset zero, the original paragraph becomes empty. At the final offset, the
new paragraph is empty.

### Join Paragraphs

```text
JoinParagraph { first, second }
```

The paragraphs must be adjacent body siblings in that order. The second
paragraph's inline content is appended to the first and normalized; the second
node is removed. Joining unrelated or reversed nodes is rejected.

## Atomic Pipeline

1. check base revision and non-empty operation list;
2. clone the current normalized document;
3. validate and apply each operation to the working document;
4. collect mapping steps and semantic inverse operations;
5. validate complete model invariants;
6. allocate the next revision;
7. reverse the collected inverse operations;
8. publish the new document, revision, map, and inverse as one commit.

No session state is changed before step 8.

## Position Mapping

Mapping steps are applied in transaction order.

| Operation | Mapping rule |
| --- | --- |
| Insert at `i`, length `n` | Offsets greater than `i` shift by `n`; an offset equal to `i` shifts only with `After` affinity. |
| Delete `[a,b)` | Offsets before/equal `a` remain; offsets inside the range collapse to `a`; offsets at/after `b` shift left by `b-a`. |
| Split node at `i` | Offsets before `i` remain on the original; offsets after `i` move to the new node at `offset-i`; equality follows affinity. |
| Join `second` into `first` at `i` | Positions in `second` move to `first` at `i+offset`; positions in `first` remain. |

Mapping a valid pre-transaction position must produce a valid
post-transaction position.

## Inverses

Each successful operation produces one semantic inverse:

| Forward | Inverse |
| --- | --- |
| Insert text | Delete inserted grapheme range |
| Delete range | Insert captured marked runs |
| Split paragraph | Join original and new paragraphs |
| Join paragraphs | Split at the captured join boundary using the removed ID |

For a multi-operation transaction, inverse operations execute in reverse order.
Inverse generation is internal and does not rely on re-running a command query.

## Undo and Redo

Session history stores forward and inverse operation lists, not public mutable
model references.

- a normal commit pushes one undo entry and clears redo;
- undo applies the entry's inverse operations as a new atomic transaction;
- redo applies the inverse generated by undo;
- remote transactions will form history boundaries when collaboration exists;
- failed undo/redo leaves both history stacks unchanged;
- empty undo/redo returns a non-fatal history error;
- history memory budgeting and persistence are later milestones.

Undo and redo receive an expected base revision so stale host actions cannot
silently apply to newer state.

## Error Mapping

| Condition | Public code |
| --- | --- |
| stale base revision | `ODC-2001` |
| unknown node, invalid boundary, invalid range, non-adjacent join | `ODC-2002` |
| empty operation/range | `ODC-2003` |
| structural control in text insertion | `ODC-2004` |
| post-application invariant failure | `ODC-2005` |
| no undo/redo history entry | `ODC-2006` |

Errors include IDs, offsets, and revisions where safe, but never surrounding
document text.

## Rejected Alternatives

**Snapshot-per-history-entry**

Rejected as the semantic design because memory scales with document size. Model
snapshots remain useful for checkpoints and recovery.

**UTF-8 or UTF-16 operation offsets**

Rejected because operation semantics must align with user-perceived caret
boundaries. Format and language bindings convert explicitly.

**Cross-paragraph delete in the first slice**

Deferred because it combines range normalization, structural join behavior,
section/table boundaries, and selection mapping. Same-paragraph deletion proves
the inverse and mapping machinery without freezing incomplete structural rules.

**Mutate in place and roll back on failure**

Rejected for Phase 0 because a working copy provides simpler atomicity. A later
persistent data structure may reduce copy cost without changing semantics.

## Acceptance Gates

- insert, delete, split, join, undo, and redo pass end-to-end SDK tests;
- marked-run deletion restores byte-identical text and mark segmentation;
- mappings cover before, boundary, inside, and after positions;
- invalid later operations prove whole-transaction atomicity;
- failed undo/redo does not consume history;
- native, MSRV, WASM, Clippy, docs, and dependency gates remain green.
