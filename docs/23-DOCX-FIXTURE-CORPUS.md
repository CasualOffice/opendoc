# DOCX Fixture Corpus Plan

**Status:** Accepted for Phase 0
**Last updated:** 2026-07-24
**Tracker:** F-009

The initial repository-owned package/security batch is implemented under
`fixtures/generated/` with SHA-256 records in `fixtures/manifest.json`.
The minimal package also drives the first package performance baseline.
Semantic, round-trip, visual, and broad performance corpus layers remain
pending.

## Purpose

The corpus is the evidence behind compatibility claims. It covers semantic
import, preservation, layout, rendering, round trip, malformed input, resource
limits, and performance.

## Source Policy

Fixtures must be:

- generated by this project and licensed under MIT;
- contributed with explicit redistribution permission;
- obtained from a source with a compatible, recorded fixture license; or
- synthesized from specifications without copying restricted sample content.

Private customer documents, sibling-repository fixtures, and vendor test files
must not be copied into this repository without explicit rights review.
Sensitive content must never be committed, even after superficial redaction.

## Layout

```text
fixtures/
├── manifest.json
├── generated/
├── compatibility/
├── malformed/
├── security/
├── performance/
├── expected/
│   ├── model/
│   ├── warnings/
│   ├── roundtrip/
│   └── render/
└── fonts/
```

Large or restricted fixtures may use a separately versioned artifact bundle.
The repository manifest records its checksum and acquisition instructions.

## Manifest Record

Each fixture records:

- stable fixture ID and path;
- source and license;
- generator and generator version when synthetic;
- SHA-256 checksum;
- feature tags;
- expected compatibility profile;
- expected warnings and errors;
- parser-limit profile;
- semantic snapshot path;
- round-trip assertions;
- visual baseline paths and font-set version;
- performance class;
- reviewer and last-reviewed date.

## Required Categories

- minimal valid package;
- text, Unicode scripts, bidi, and grapheme edge cases;
- paragraph and character styles;
- lists and numbering;
- tables, spans, borders, and page splits;
- sections, headers, footers, columns, and page numbering;
- images, anchors, wrapping, and relationships;
- comments, bookmarks, fields, notes, and tracked changes;
- unknown namespaces and safely preservable parts;
- malformed ZIP, XML, relationship, and content-type data;
- every parser-limit boundary;
- large text-, table-, image-, and review-heavy documents.

## Comparison Layers

Each compatible fixture can have four independent expectations:

1. package: required parts, relationships, and security outcome;
2. semantic: normalized model and warning set;
3. round trip: retained, changed, flattened, or dropped features;
4. visual: structural geometry and perceptual image comparison.

Passing one layer does not imply another. A document can import semantically but
fail round-trip preservation, or preserve content without rendering it.

## Golden Update Policy

Golden files are never updated solely to make tests green. An update requires:

- a linked behavior or design change;
- a human-readable diff summary;
- compatibility impact classification;
- visual inspection where rendering applies;
- reviewer acknowledgement for data-loss or pagination changes.

## First Corpus Batch

Before the first DOCX parser merge, create:

- one minimal valid blank DOCX;
- one paragraph with mixed Unicode and marks;
- one relationship/path traversal rejection case;
- one high-expansion ZIP rejection case;
- one unknown safe XML preservation case;
- one deterministic package-order fixture.

The generator source belongs in the repository so fixtures can be reproduced.
