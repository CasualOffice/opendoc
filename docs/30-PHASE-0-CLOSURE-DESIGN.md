# Phase 0 Closure Design

**Status:** Accepted for implementation
**Decision date:** 2026-07-24
**Tracker:** P0-008
**Implementation:** Complete

## Outcome

Close Phase 0 with auditable foundation evidence while preserving the roadmap's
capability order and the separation between this repository and the sibling
study repository.

Phase 0 will be complete when the repository contains:

- the implemented foundation runtime and package reader;
- a repository-owned package/security fixture batch;
- package-level conformance tests;
- reproducible performance reports for implemented paths;
- a bounded package-reader fuzz target with pull-request build and scheduled
  execution;
- a versioned exit report that records every deliverable as passed, deferred to
  its owning capability phase, or explicitly out of scope.

## Roadmap Correction

The original Phase 0 wording asked for round-trip and rendering snapshots before
the roadmap introduces a DOCX writer or renderer. It also described importing a
fixture corpus from the current Casual Docs implementation even though that
repository is a separate study/reference workspace.

The corrected sequencing is:

- Phase 0 owns report schemas, update policy, package/model performance
  baselines, synthetic package fixtures, and an explicit baseline readiness
  report;
- Phase 1D owns visual baselines when the reference renderer exists;
- Phase 2 owns semantic DOCX round-trip baselines when the writer exists;
- fixtures from another repository are never copied without explicit rights
  review and an intentional cross-repository decision.

This correction does not lower a product compatibility gate. It prevents
placeholder artifacts from being mistaken for implementation evidence.

## Corpus Closure

The generated Phase 0 batch adds:

- mixed Unicode WordprocessingML package content, including combining marks,
  right-to-left text, and an emoji sequence;
- an unknown safe package part that can be enumerated and read byte-for-byte;
- existing valid, traversal, expansion, duplicate, malformed, and deterministic
  ordering cases.

The package reader does not claim to interpret the XML. Tests prove package
admission, deterministic metadata, rejection policy, and exact safe-part reads.
Semantic interpretation and preservation-through-save remain later gates.

Every fixture remains:

- generated by source committed in this repository;
- Apache-2.0 licensed;
- listed in `fixtures/manifest.json`;
- protected by a SHA-256 CI check;
- free of copied customer, vendor, or sibling-repository content.

## Fuzz Boundary

Use `cargo-fuzz` 0.13.2 with exactly `libfuzzer-sys` 0.4.13 in an independent
`fuzz/` workspace. The fuzz dependency graph has its own committed lockfile so
it does not alter product or MSRV dependencies.

The initial `docx_package` target accepts arbitrary bytes and:

1. admits them under a 1 MiB input limit and lower expanded-data/entry limits;
2. returns normally for every typed rejection;
3. enumerates every admitted regular part;
4. reads each admitted part and verifies the actual length against metadata;
5. never writes package content to disk.

The target is bounded to prevent a valid but adversarial sample from turning a
short CI campaign into uncontrolled resource consumption.

## Fuzz CI

Pull requests run a build-only `fuzz-build` job on pinned nightly Rust. This
proves that the target and instrumentation remain compilable without adding
timing variability to the required matrix.

The scheduled security workflow runs the target with:

- generated fixtures as initial seeds;
- 60 seconds maximum campaign time;
- 5 seconds maximum per input;
- 1 MiB maximum generated input;
- a 2 GiB process RSS limit.

A crash, timeout, sanitizer failure, or panic fails the workflow. Reproducing
inputs are reviewed, minimized, and committed as regression fixtures when they
are safe and non-sensitive.

The official Rust Fuzz Book and `cargo-fuzz` repository were checked on
2026-07-24:

- [cargo-fuzz guide](https://rust-fuzz.github.io/book/cargo-fuzz/guide.html);
- [cargo-fuzz repository](https://github.com/rust-fuzz/cargo-fuzz).

## Exit Evidence

`31-PHASE-0-EXIT-REPORT.md` will be the authoritative closure record. It must
link each roadmap item and exit gate to concrete files, tests, CI jobs, reports,
or ADRs.

The report cannot mark a capability as passed based only on a design. Deferred
visual and round-trip artifacts must name their owning future phase and cannot
be counted as current runtime support.

## Rejected Alternatives

### Copy study-repository fixtures

Rejected because the user established that repository as separate ongoing work,
and provenance or redistribution rights are not established here.

### Generate placeholder render screenshots

Rejected because no OpenDoc layout or renderer exists. Such images would test a
different renderer and create false fidelity evidence.

### Treat package byte copying as DOCX round trip

Rejected because a no-op copy does not exercise semantic import, editing, or
serialization.

### Run time-bounded fuzzing on every pull request

Rejected because fuzz outcomes depend on scheduling and random exploration.
Pull requests compile the target; scheduled CI performs the campaign.

## Acceptance Gates

- roadmap sequencing is corrected through ADR-023;
- corpus isolation and source policy remain explicit;
- mixed-Unicode and unknown-safe-part fixtures are generated and checksummed;
- package tests verify both fixtures without semantic-import claims;
- fuzz target uses bounded package limits and a committed lockfile;
- pull requests compile the fuzz target;
- scheduled security CI runs the seeded campaign;
- dependency policy and audit cover the fuzz workspace;
- Phase 0 exit report links all evidence and names future-phase deferrals;
- all native, Windows, macOS, WASM, MSRV, docs, lint, audit, policy,
  benchmark-smoke, and fuzz-build checks pass.
